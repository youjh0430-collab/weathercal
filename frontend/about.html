날씨를 입힌 캘린더, 하루를 더 똑똑하게

서비스 개요

WeatherCal은 월간 캘린더에 날씨를 입혀, 기존 일정에는 날씨 경고를, 빈 날에는 활동 추천을 해주는 서비스입니다.

🌦
날씨는 메인이 아닌 가중치

날씨가 주인공이 아니라, 내 일정을 더 잘 판단하게 돕는 보조 역할

⚠️
야외 일정 날씨 경고

비 예보, 한파, 폭염 등 야외 일정에 영향을 주는 날씨를 미리 알려줌

💡
빈 날 활동 추천

일정이 없는 날, 날씨와 기온에 맞는 활동을 제안 (한국인 맞춤)

👤
게스트 모드 지원

로그인 없이 바로 사용 가능. 로그인하면 사용자별 일정 분리

개발 순서

1

백엔드 기초

FastAPI 프로젝트 설정, SQLite DB 스키마 설계, Pydantic 모델 정의

2

핵심 API 구현

일정 CRUD, OpenWeatherMap 연동, 날씨 캐시 전략, 경고/추천 서비스

3

프론트엔드 구현

월간 캘린더 렌더링, 브리핑 카드, 일정 모달, 키워드 자동 분류

4

기상청 API 전환

OpenWeatherMap → 기상청 ASOS(과거) + 단기예보 + 중기예보 3개 API 통합, 전국 16개 도시 지원

5

UI 개편 + PWA

앱 스타일 레이아웃, PC/모바일 반응형, 날씨별 테마 배경, 홈 화면 설치 지원

6

소셜 로그인 + 사용자 분리

Google/카카오 OAuth, 사용자별 일정 분리, 게스트 모드 지원

7

배포 + 디버깅

Render 배포, OAuth 환경변수 설정, 배포 환경 디버그 엔드포인트 추가

주요 의사결정

날씨를 서비스의 어디에 둘 것인가?
날씨를 메인이 아닌 가중치(Sub)로 활용. 기존 날씨앱과 차별화하기 위해, 날씨 자체를 보여주기보다 내 일정에 "영향을 주는 정보"로 포지셔닝함.
왜 OpenWeatherMap에서 기상청 API로 바꿨나?
OpenWeatherMap 무료 플랜은 5일 예보만 가능하고 과거 날씨를 조회할 수 없음. 기상청 API는 과거(ASOS) + 단기(3일) + 중기(10일) 모두 무료로 제공해서 데이터 범위가 훨씬 넓어짐.
왜 바닐라 JS로 만들었나?
React/Vue 같은 프레임워크를 쓰면 학습 비용이 추가됨. PM 입장에서 서버와 DB를 이해하는 것이 목표이므로, 프론트엔드는 최대한 단순하게 유지.
게스트 모드를 왜 추가했나?
로그인이 필수이면 서비스 진입 장벽이 높아짐. "먼저 써보고, 마음에 들면 로그인"하는 흐름이 자연스러워서 게스트 모드를 지원.
SQLite를 쓴 이유는?
학습 목적의 과제이므로, 별도 설치 없이 Python 내장 모듈로 바로 사용할 수 있는 SQLite가 적합. 다만 Render 배포 시 재배포마다 DB가 초기화되는 한계가 있음.
AI 도구를 어떻게 활용했나?
비개발자(PM)가 Claude Code를 활용하여 개발을 진행. 아이디어 구체화 → 설계 문서 작성 → 구현 계획 → 단계별 실행 → 코드 리뷰 → 검증의 6단계 워크플로우를 따름.
